# 测试数据分析智能体系统技术设计文档（对齐重构版）

## 1. 文档目标

本文档用于将 `test-analyst` 的技术设计与新版项目需求书保持一致。

本次设计重心不是定义一个“多智能体节点清单”，而是明确系统如何真正支持：

- 复杂测试数据理解
- 探索式分析
- 多轮修正与上下文继承
- 确定性执行与结果可追溯

设计目标是让系统最终表现为一个面向真实测试工作的分析助手，而不是只能跑样例的编排流程。

## 2. 设计原则

系统技术设计遵循以下原则：

1. 对用户呈现为持续对话的分析助手，而不是一次性表单任务。
2. `LLM` 负责理解需求、识别修订语义、规划任务和生成澄清，不直接生成最终数值结果。
3. 所有数据读取、筛选、聚合、统计、导出必须由确定性程序执行。
4. 规则配置、字段别名、模板配置和业务公式是辅助知识，不是主流程引擎。
5. 系统必须优先支持复杂和未知的数据结构，而不是只优化样例路径。
6. 每次字段映射、工具调用、公式选择和结果输出都必须可追溯、可复核。
7. 是否采用多智能体是实现策略，不是技术设计的唯一前提。

## 3. 总体目标架构

系统推荐采用四层结构：

1. 交互与会话层
2. 语义理解与任务规划层
3. 数据执行与交付层
4. 配置、审计与存储层

四层职责分别如下。

### 3.1 交互与会话层

负责：

- 接收文件和用户输入
- 展示结构画像、澄清问题、执行进度和结果摘要
- 管理会话和版本
- 接收用户后续补充、否定和修正

### 3.2 语义理解与任务规划层

负责：

- 识别用户当前意图
- 继承会话上下文
- 发现缺失信息并发起澄清
- 将自然语言需求转成结构化分析任务

### 3.3 数据执行与交付层

负责：

- 文件读取和数据画像
- 字段标准化和候选映射
- 筛选、聚合、统计、对比、异常检测
- 结果导出、图表生成和报告生成

### 3.4 配置、审计与存储层

负责：

- 提供字段别名、模板配置、业务规则和权限配置
- 记录会话、任务版本、审计日志和产物元数据
- 支撑历史回看、版本对比和问题追踪

## 4. 关键设计结论

### 4.1 不将“多智能体”写死为主架构

系统可以采用：

- 单智能体 + 工具调用
- 多智能体协作
- 状态图编排
- 混合式工作流

但不论哪种实现，必须满足以下技术目标：

1. 能处理复杂未知数据。
2. 能支持多轮修正。
3. 能输出结构化且可审计的结果。
4. 不把规则链当作主要理解手段。

### 4.2 将“先理解数据，再执行分析”写入主流程

运行时主流程不应是“解析需求后直接套模板导出”，而应为：

1. 文件接入
2. 数据画像
3. 需求理解
4. 上下文继承
5. 澄清收敛
6. 任务规划
7. 确定性执行
8. 校验与解释
9. 导出与继续修订

### 4.3 将“复杂数据适配”作为一等能力

系统要优先适配以下情况：

- 多 sheet
- 多表头
- 宽表、长表、交叉表
- 字段命名不稳定
- 指标组和字段范围表达
- 模板不完全匹配

这部分不能依赖少量样例和 if-else 规则勉强支撑。

## 5. 核心运行流程

### 5.1 首轮任务流程

首轮任务建议按以下阶段执行：

1. 接收用户上传文件与自然语言需求。
2. 读取文件并生成结构画像。
3. 基于画像和会话上下文解析用户意图。
4. 判断是否存在关键缺口。
5. 如有缺口，发起少量关键澄清。
6. 形成结构化任务计划。
7. 调用确定性工具执行。
8. 输出结果、风险和审计信息。

### 5.2 多轮修订流程

当用户继续输入时，运行时应先判断输入类型：

- 新任务
- 补充说明
- 修正口径
- 否定前一轮结果
- 导出要求变更

之后执行：

1. 加载当前会话已确认上下文。
2. 识别哪些参数沿用，哪些参数更新。
3. 只重算受影响的分析步骤。
4. 生成新版本结果并记录差异。

### 5.3 澄清流程

澄清流程必须满足：

1. 只针对阻塞任务推进的关键问题提问。
2. 优先一次问 1 到 3 个高价值问题。
3. 能结合候选项提问，而不是让用户从零描述。
4. 用户回答后可继续执行，而不是重新开始整轮任务。

## 6. 核心状态模型

建议系统维护以下核心状态对象：

```text
ConversationState
  session_id
  task_id
  parent_task_id
  user_message
  conversation_summary
  confirmed_context
  file_contexts
  data_profile
  requirement_spec
  clarification_state
  execution_plan
  analysis_result
  validation_result
  artifacts
  warnings
  trace_events
```

### 6.1 `confirmed_context`

应保存已经明确确认过的上下文，例如：

- 当前使用的文件
- source sheet
- 有效数据区
- 关键字段映射
- 已选测试项或指标组
- 分组维度
- 公式口径
- 输出格式
- 模板信息

### 6.2 `data_profile`

用于表达系统对输入文件的“理解结果”，建议至少包含：

- 文件类型
- sheet 列表
- 候选表头位置
- 行列规模
- 字段标准化结果
- 候选业务语义映射
- 风险与异常提示

### 6.3 `requirement_spec`

用于表达当前轮的结构化任务需求，建议包含：

```text
RequirementSpec
  raw_request
  intent_type
  target_files
  source_sheet
  data_region
  requested_metrics
  metric_groups
  filters
  aggregate_by_fields
  compare_dimensions
  requested_statistics
  derived_metrics
  formula_config
  output_formats
  template_sheet_name
  clarification_questions
  inherited_context_notes
  parsed_notes
```

重点说明：

- `intent_type` 用于识别当前输入是新建、补充、修订、否定、追问还是仅导出。
- `metric_groups` 用于表达字段家族或字段范围。
- `formula_config` 不能只存一个衰减方法名，应为后续扩展派生公式留足空间。

## 7. 语义理解与规划设计

### 7.1 需求理解职责

语义层必须能够识别：

- 用户要分析什么
- 数据可能来自哪里
- 要按什么维度筛选、统计、对比
- 本轮输入相对于上轮是延续还是修正
- 哪些信息缺失且会阻塞执行

### 7.2 需求理解的输入

需求理解模块输入建议包括：

- 当前用户消息
- 会话摘要
- 已确认上下文
- 数据画像摘要
- 规则配置摘要

### 7.3 需求理解的输出

输出至少包括：

- 结构化需求对象
- 是否需要澄清
- 澄清问题列表
- 本轮更新点
- 对上轮上下文的继承说明

### 7.4 设计约束

1. 不能只靠关键词枚举来决定用户真实意图。
2. 不能把全部历史原文直接拼接给模型让模型自行猜测。
3. 应优先使用结构化上下文和摘要上下文，减少上下文污染。
4. 对高风险字段和公式必须经过确认或有明确证据支持。

## 8. 数据画像与字段映射设计

### 8.1 数据画像职责

数据画像模块负责回答：

- 哪些 sheet 最可能是主数据
- 哪一行最可能是表头
- 哪些字段疑似维度字段
- 哪些字段疑似测试项或指标列
- 数据中有哪些明显风险

### 8.2 字段映射策略

字段映射建议分三层：

1. 规范化层
   统一大小写、空格、特殊字符、换行和噪声前后缀。
2. 候选识别层
   结合字段名语义、值分布特征、历史映射和别名配置生成候选。
3. 确认层
   对高置信候选直接采用，对中低置信候选触发澄清。

### 8.3 指标组识别

系统应支持将以下表达识别为一类分析对象：

- `MTFH_1~MTFH_17`
- `RCLvUnif_*`
- “所有亮度相关项”
- “这组 MTF 点位”

技术上可结合：

- 模式匹配
- 前缀聚类
- 列名向量或语义相似度
- 数据画像中的字段统计信息

但最终仍需形成明确可执行的字段集合。

## 9. 数据执行层设计

### 9.1 执行层职责

执行层只负责确定性工作，包括：

- 数据读取
- 类型转换
- 数据清洗
- 字段筛选
- 分组聚合
- 统计计算
- 派生指标计算
- 异常检测
- 结果导出

### 9.2 执行层能力模块

建议拆成以下能力模块：

1. 文件读取模块
2. 数据标准化模块
3. 字段解析与展开模块
4. 分析算子模块
5. 模板导出模块
6. 图表与报告模块
7. 校验与审计模块

### 9.3 分析算子要求

分析算子至少应支持：

- 筛选
- 去重
- 分组统计
- 最值和均值统计
- 缺失值和异常值检测
- 多阶段对比
- 变化率和衰减率计算
- 指标组并行统计

### 9.4 公式执行要求

1. 业务公式不得隐式写死在提示词中。
2. 公式必须以结构化参数传入执行层。
3. 当公式不明确时，执行层不得默认猜测。
4. 公式版本和参数必须进入审计日志。

## 10. 输出层设计

### 10.1 输出原则

输出层应优先满足“可交付”和“可解释”。

结果不只是一份文件，还应包含：

- 结果摘要
- 统计口径说明
- 风险和不确定项
- 产物路径

### 10.2 输出类型

当前建议支持：

- Excel
- 图表
- Markdown
- 可选 PDF / Docx

### 10.3 模板兼容策略

模板是增强项，不是前置依赖。

设计要求：

1. 有模板时优先匹配模板。
2. 模板不完全匹配时，输出差异说明。
3. 不能因为模板不匹配就完全无法输出结果。

## 11. 校验与审计设计

### 11.1 校验职责

校验层负责确认：

- 用户要求的分析对象是否被覆盖
- 输出字段是否齐全
- 使用的字段映射是否一致
- 使用的公式是否符合当前口径
- 是否存在需人工确认的高风险项

### 11.2 审计内容

系统至少需要记录：

1. 原始用户输入
2. 当前会话摘要
3. 已继承上下文
4. 文件画像结果
5. 字段映射关系
6. 执行计划
7. 工具调用参数和返回摘要
8. 公式配置
9. 输出产物路径
10. 警告、异常和人工确认点

### 11.3 可观测性要求

系统应能支持查看：

- 任务版本
- 修订历史
- 关键状态变化
- 分析结果与审计结果之间的对应关系

## 12. 回退与失败策略

### 12.1 `LLM` 不可用

当 `LLM` 不可用时：

1. 可以退回到有限的规则兼容模式。
2. 必须明确告知用户当前处于降级状态。
3. 降级模式只作为兜底，不应成为默认主路径。

### 12.2 字段映射不确定

当候选字段存在歧义时：

1. 先给出候选项和判断依据。
2. 等待用户确认后再执行。
3. 不允许低置信度静默继续。

### 12.3 公式或口径不明确

当衰减、变化率、达标等业务口径不明确时：

1. 先追问。
2. 不允许默认假设。
3. 不允许输出看似完整但依据不足的业务结论。

### 12.4 模板不兼容

当模板不兼容时：

1. 给出不兼容说明。
2. 输出标准结果文件。
3. 保留后续人工调整接口。

## 13. 与当前代码结构的对齐建议

结合当前项目结构，建议职责大致对齐如下：

- `file_loader.py`
  负责文件读取和基础结构提取
- `requirement_parser.py`
  负责需求结构化，但应继续弱化硬编码规则占比，增强上下文继承和复杂表达支持
- `analysis.py`
  负责确定性分析算子
- `reporting.py`
  负责结果导出、图表和报告生成
- `storage.py`
  负责会话、版本、审计和产物元数据
- `workflow.py` / `langgraph_workflow.py`
  负责运行时编排，但不应把当前节点设计误当成产品边界

这意味着后续演进重点应是：

1. 强化 `data_profile` 与字段语义识别能力。
2. 强化 `RequirementSpec` 对复杂任务的表达能力。
3. 强化多轮修订时的局部重算和上下文继承。
4. 将模板导出、审计和结果解释与执行层更紧密地绑定。

## 14. 分阶段落地建议

### 第一阶段

优先完成：

- 文件画像
- 会话上下文继承
- 结构化需求对象升级
- 通用分析算子稳定化

### 第二阶段

补强：

- 字段组识别
- 复杂表结构适配
- 模板兼容输出
- 审计和差异化版本管理

### 第三阶段

再推进：

- 更丰富的复杂指标公式
- 更强的异常检测
- 更细的交互澄清策略
- 私有化部署能力

## 15. 结论

`test-analyst` 的技术设计不应继续围绕“如何把样例流程编排得更完整”展开，而应围绕以下问题持续演进：

- 如何让系统更好地理解复杂测试数据
- 如何让系统更好地承接自然语言分析需求
- 如何让系统在多轮修订中持续稳定工作
- 如何让每个分析结果都可解释、可复核、可交付

只有这样，系统才会从“演示型项目”真正变成“可用的测试数据分析效率工具”。
