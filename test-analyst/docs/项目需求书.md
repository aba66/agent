# 测试数据分析智能体系统项目需求书（重构版）

## 1. 文档目的

本文档用于重新定义 `test-analyst` 项目的产品需求，目标不是做一个“能跑通样例的多智能体编排演示”，而是建设一个真正能提升测试人员处理复杂测试数据效率的分析系统。

本次重构重点解决以下问题：

1. 现有需求过度强调智能体角色拆分和流程编排，容易把实现引向“固定流程 + 固定样例 + 固定字段”的系统。
2. 对复杂数据场景的描述不足，没有充分约束系统必须具备“理解未知结构、支持探索分析、允许多轮收敛”的能力。
3. 数据保密不是当前阶段的首要矛盾，当前更重要的是分析效率、适配复杂数据的能力、结果可复核性和可持续迭代能力。

## 2. 项目背景

测试人员需要频繁分析产品测试数据，以判断：

- 产品是否达到预期
- 是否存在异常或退化
- 是否满足进入下一阶段的条件
- 哪些问题需要进一步排查

现实中的测试数据通常并不规整，常见问题包括：

- 文件格式不统一，既可能是 `xlsx/xls/csv`，也可能包含多 sheet、多表头、多段数据区
- 字段命名不统一，同一指标在不同项目、不同人员导出的表中名字不同
- 测试项数量多，且经常以“字段族”“点位范围”“某类指标集合”的方式出现
- 用户需求不是固定模板，而是边分析边修正，经常包含补充、否定、对比、追问
- 分析结论必须能回溯数据来源和计算过程，否则无法用于汇报或决策

因此，本项目要解决的不是“把一份样例 Excel 套进一个脚本”，而是要构建一个面向真实测试分析工作的对话式分析系统。

## 3. 当前阶段定位

### 3.1 核心定位

本项目当前阶段定位为：

- 一个以“文件上传 + 自然语言需求 + 多轮修正”为核心交互方式的测试数据分析助手
- 一个面向复杂、非标准化测试数据的分析工作台
- 一个能把“理解问题、识别数据、执行分析、输出结果、继续修正”串成闭环的系统

### 3.2 当前阶段优先级

当前阶段优先级如下：

1. 优先提升复杂测试数据的处理效率
2. 优先支持未知结构和非标准字段的分析任务
3. 优先保证结果可解释、可追溯、可复核
4. 优先支持多轮对话式修订

### 3.3 当前阶段暂不作为主目标

当前阶段不把以下事项作为主目标：

1. 不把“本地私有部署”和“绝对不出网”作为首要验收目标
2. 不把“多智能体编排本身”作为产品价值
3. 不把“固定模板导出”作为唯一核心能力
4. 不追求替代完整 BI 平台、数据仓库或企业级报表系统

说明：

- 出于经费和资源限制，当前阶段允许接入公开大模型 API。
- 系统应为后续私有化部署预留接口，但本期验收重点不是数据安全，而是效率和能力。

## 4. 产品目标

### 4.1 总体目标

系统应帮助测试人员把原本需要手工完成的以下工作显著提效：

- 理解测试表结构
- 定位目标字段和测试项
- 清洗和标准化数据
- 完成筛选、分组、聚合、对比、异常识别和派生指标计算
- 输出可交付的表格、图表和分析结论
- 在分析过程中持续修正需求，而不是每次重来

### 4.2 成功标准

如果系统成功，用户应获得以下体验：

1. 面对陌生或复杂测试数据时，系统能先帮用户“看懂数据”。
2. 用户不必先把需求翻译成固定脚本参数，只需自然描述任务。
3. 当需求不完整时，系统能提出少量但关键的澄清问题。
4. 当用户修正分析口径时，系统能在当前上下文上继续，而不是从头来。
5. 输出结果不仅有数字，还有字段映射、公式口径、异常说明和来源说明。

## 5. 产品边界

### 5.1 本系统是什么

本系统是一个面向测试数据分析的任务型智能体产品。它接收：

- 一个或多个测试数据文件
- 用户的自然语言分析需求
- 可选的模板、规则、业务口径或人工补充信息

它输出：

- 结构化分析结果
- 图表和报告
- 过程说明、风险提示和审计信息

### 5.2 本系统不是什么

本系统不是：

1. 只能处理样例字段和样例模板的脚本编排器
2. 靠硬编码关键词命中才可工作的半自动 Excel 工具
3. 只负责“生成一份固定报表”的模板套壳器
4. 以“展示多智能体流程图”为目标的技术演示项目

### 5.3 对“多智能体”的要求

是否采用多智能体，是实现方案，不是产品目标。

需求层面只要求系统具备以下能力：

- 能理解用户意图
- 能维护上下文
- 能规划并执行分析任务
- 能校验结果并继续对话

至于这些能力最终由单智能体、多个智能体、状态图还是其他运行时实现，不应在需求书中被写死。

## 6. 目标用户

主要用户包括：

- 测试工程师：上传数据并提出分析需求
- 测试分析人员：做指标汇总、阶段对比、异常定位、报告输出
- 项目负责人或质量负责人：查看结果、关注结论和风险项

## 7. 典型使用场景

系统至少应支持以下真实场景，而不是只支持样例场景：

1. 用户上传一个结构不熟悉的 Excel，先让系统识别 sheet、表头、字段、候选数据区，再继续分析。
2. 用户按测试项名称、字段族、前缀范围或自然语言描述来指定分析对象，例如：
   - `MTFH_1~MTFH_17`
   - “所有亮度相关项”
   - “这一组对比度指标”
3. 用户要求按不同维度进行汇总，例如按 `SN`、阶段、测试节点、机型、批次分别统计。
4. 用户要求计算项目特有指标，例如衰减率、变化率、达标率、上下阶段差异。
5. 用户需要对多文件、多批次、多阶段数据做横向比较。
6. 用户发现结果口径不对，要求在原任务基础上继续修正。
7. 用户要求输出 Excel、图表、文字总结，必要时套用模板，但模板不是唯一输出方式。
8. 用户面对字段缺失、命名混乱、单位不一致、表头多层等复杂情况时，系统仍能推进任务，而不是直接失败。

## 8. 核心产品理念

### 8.1 系统首先要“理解数据”

系统的第一职责不是立刻套流程，而是先帮助用户建立数据认知，包括：

- 文件里有哪些 sheet
- 哪些区域像有效数据
- 表头从哪一行开始
- 哪些字段可能是测试项、阶段、SN、测试节点、时间、备注、机型等
- 哪些字段存在缺失、重复、单位混杂、异常命名

### 8.2 系统要支持“探索式分析”

真实分析任务往往不是一次就能描述完整，因此系统必须支持：

- 先概览再深入
- 先找字段再定指标
- 先出初版再修口径
- 先定位异常再补充解释

系统不能把用户每次输入都当成新的独立任务，而应把它理解为当前分析过程中的增量操作。

### 8.3 系统要支持“复杂而未知”的数据

需求书必须明确要求：

1. 系统不能依赖少量预设字段名才能工作。
2. 系统不能只在样例文件结构下有效。
3. 只要数据是表格类且逻辑上可解析，系统就应尽可能通过识别、澄清和人工确认机制推进任务。
4. 规则配置、字段别名、模板配置只能提升成功率，不能成为系统可用性的前提条件。

### 8.4 系统必须“可复核”

系统给出的结果必须能回答以下问题：

- 用了哪个 sheet
- 哪些列被映射成什么含义
- 采用了什么公式和口径
- 哪些数据被过滤、填充、聚合或判定为异常
- 哪些地方存在不确定性，需要人工确认

## 9. 核心业务闭环

系统的标准闭环应为：

1. 接收文件和用户需求
2. 识别并展示数据结构摘要
3. 解析用户意图并结合会话上下文形成分析目标
4. 发现信息缺口并提出必要澄清
5. 制定分析计划并调用确定性工具执行
6. 输出结果、证据和风险提示
7. 接收用户追问或修正并继续迭代

其中第 2 到第 7 步必须支持循环，不应被设计成一次性流程。

## 10. 功能性需求

### 10.1 文件接入与数据画像

系统应支持：

1. 上传一个或多个 `xlsx/xls/xlsm/csv` 文件。
2. 识别多 sheet、多表头、多段数据区场景。
3. 输出文件结构摘要，包括文件名、sheet 列表、候选数据区、行列规模、疑似表头位置。
4. 识别常见数据问题，包括空行空列、重复列名、单位混杂、合并单元格、乱码、公式单元格、半结构化说明区。
5. 在无法确定有效数据区时，引导用户指定 sheet、表头行或数据范围。
6. 原始文件默认只读，结果输出不得静默覆盖原文件。

### 10.2 字段理解与语义映射

系统应支持：

1. 对字段名进行标准化处理，例如去空格、换行、隐藏字符、前后缀噪声。
2. 识别字段语义，而不仅是字段字面值，例如识别“SN”“样机编号”“设备号”可能属于同一语义。
3. 对测试项、阶段、测试节点、时间、批次、机型、备注等常见业务字段建立候选映射。
4. 对字段族、指标组、前缀范围、编号区间做识别，例如：
   - `MTFH_1~17`
   - `RCLvUnif_*`
   - “所有色温相关字段”
5. 当映射不确定时，必须给出候选项和置信说明，等待用户确认。
6. 映射成功后，应记录来源字段和标准语义。

### 10.3 需求理解与会话继承

系统应支持从自然语言中持续理解以下内容：

- 要分析哪些测试项或指标组
- 数据来自哪个文件、哪个 sheet、哪个数据区
- 需要按哪些维度筛选、分组、对比
- 要计算哪些指标或派生值
- 要输出什么形式的结果
- 本轮输入是新需求、补充、修正、否定还是追问

系统必须支持以下能力：

1. 在同一会话中继承已确认上下文，例如已选 sheet、字段映射、模板、统计口径。
2. 正确理解增量修订，而不是每轮重置需求。
3. 当用户说“沿用刚才的口径，只改衰减公式”时，只修改对应部分。
4. 当用户说“这个结果不对，要按 SN 分开统计”时，系统应在原任务基础上重新执行。

### 10.4 澄清机制

系统应只在关键缺口存在时发起澄清，避免一次性抛出过多问题。

以下情况应触发澄清：

1. 有多个候选 sheet 或数据区，且无法确定用户目标。
2. 目标测试项名称模糊，存在多个近似字段。
3. 用户要求计算业务指标，但公式口径未定义。
4. 用户要求输出模板化结果，但模板 sheet 或映射规则不明确。
5. 分组维度、过滤条件或时间范围缺失，导致结果口径不唯一。

### 10.5 数据分析能力

系统应支持以下通用分析能力：

1. 按字段值筛选数据。
2. 按一个或多个维度分组统计。
3. 计算基础统计量，如计数、均值、最大值、最小值、标准差、中位数。
4. 计算变化类指标，如差值、变化率、衰减值、环比、同比、阶段差异。
5. 支持多指标并行分析和结果并排展示。
6. 支持跨文件、跨批次、跨阶段对比。
7. 识别异常值、缺失值、重复值、离群点和可疑数据模式。
8. 允许基于用户描述临时生成派生列或中间指标。
9. 支持“先明细后汇总”“先汇总后下钻”的分析过程。

特别要求：

1. 业务指标公式不能默认写死。
2. 公式可以来自配置、用户输入或人工确认。
3. 在公式不明确时，系统不得自作主张输出看似准确的结论。

### 10.6 复杂数据适配能力

这是本项目区别于“样例脚本系统”的关键要求。

系统必须支持以下能力：

1. 面对未在配置中预置过的新字段，仍能基于语义和数据特征完成候选识别。
2. 面对列名变化、列顺序变化、sheet 名变化，只要业务含义仍可识别，任务就不应直接失效。
3. 面对字段缺失时，能够明确指出缺什么、影响什么，而不是直接失败或静默跳过。
4. 面对宽表、长表、交叉表、半结构化表格，能够先做结构判断，再决定处理方式。
5. 面对样例之外的新分析诉求，只要属于表格数据分析范畴，系统应尽量通过规划和工具组合完成，而不是提示“不支持”。

### 10.7 输出与交付物

系统应支持输出：

1. Excel 结果文件
2. 图表，如柱状图、折线图、散点图、箱线图
3. Markdown 报告
4. 可选的 PDF 或 Docx 报告
5. 审计日志或分析摘要

输出结果应满足：

1. 默认不覆盖原始输入文件。
2. 结果中应包含必要的字段名、单位、时间、统计口径说明。
3. 若存在不确定项，应明确标注“需人工确认”。
4. 若用户提供模板，应优先兼容模板；若模板不适配，也应能输出标准结果，而不是完全依赖模板。

### 10.8 解释、追溯与审计

系统必须记录并可展示：

1. 输入文件与使用的 sheet
2. 关键字段映射关系
3. 使用的分析维度、过滤条件和公式口径
4. 中间步骤中的警告和异常
5. 输出文件路径和生成时间
6. 用户在多轮会话中的关键修订点

## 11. 非功能性需求

### 11.1 准确性与可信性

1. 系统不得编造不存在的数据或字段。
2. 当字段含义、公式口径或结果解释不确定时，必须显式提示。
3. 数值计算必须由确定性程序执行，不能由模型直接“算结果”。
4. 结果中应保留足够证据，便于人工抽查。

### 11.2 易用性

1. 用户应能通过接近自然对话的方式完成任务，而不是学习复杂参数体系。
2. 系统应尽量减少重复询问已经确认过的信息。
3. 错误提示要指向问题本身，例如“未找到字段”“sheet 名不匹配”“公式未定义”，而不是输出笼统失败。

### 11.3 性能

当前阶段建议验收目标：

- 对 50MB 以内的常见测试文件，2 分钟内完成结构识别和初步画像
- 对常见单文件分析任务，5 分钟内输出首版结果
- 对多轮修订任务，能复用上下文和中间结果，减少重复执行时间

### 11.4 可扩展性

系统后续应易于扩展：

- 新文件格式
- 新分析算子
- 新图表类型
- 新模板
- 新模型供应商
- 新规则配置
- 新部署模式，包括后续私有化部署

## 12. 实现约束与原则

### 12.1 需求层面的强约束

1. 不允许把项目实现成“样例驱动的固定流程系统”。
2. 不允许把字段别名表、关键词表、if-else 规则链当成主能力来源。
3. 不允许系统只在少量演示文件上表现良好，而在字段变化后大面积失效。
4. 不允许把“能导出模板 Excel”误当成“完成复杂数据分析”的替代品。

### 12.2 允许的实现方式

实现上可以采用但不限于以下方式：

- 单智能体 + 工具调用
- 多智能体协作
- 状态图编排
- 工作流引擎
- 规则配置 + LLM 混合模式

前提是：

- 主价值必须体现在复杂数据分析能力和多轮交互能力上
- 工具执行必须是确定性的
- 关键决策必须可追溯

## 13. 典型用户故事

### 用户故事 1：陌生数据理解

作为测试工程师，我上传一个自己也不完全熟悉的测试 Excel，希望系统先告诉我：

- 哪个 sheet 最可能是主数据
- 哪些列像测试项、阶段、SN、测试节点
- 哪些地方存在脏数据和风险

这样我才知道后续怎么提分析需求。

### 用户故事 2：复杂指标组分析

作为测试分析人员，我希望直接说“把 `MTFH_1~MTFH_17` 和 `MTFV_1~MTFV_17` 按阶段和 SN 分别统计最小值、最大值、平均值和衰减率”，系统就能把这句话转成实际分析任务，而不是要求我逐列手工配置。

### 用户故事 3：多轮修正

作为测试分析人员，我希望在拿到第一版结果后继续说：

- “不要整表统计，按每个测试项目分别统计”
- “保留刚才的维度，只把衰减口径改成相对首条记录”
- “把异常样机单独拉出来”

系统应在当前上下文上继续，而不是让我从头描述。

### 用户故事 4：结果可交付

作为项目负责人，我不只想看到结论，还希望结果中能说明：

- 数据来自哪里
- 用了什么规则
- 哪些数据异常
- 哪些地方需要人工确认

这样结果才能用于汇报和决策。

## 14. 验收标准

以以下标准验收本期需求：

1. 系统能处理非样例文件，而不是只能处理预先演示过的数据结构。
2. 用户上传文件后，系统能先输出结构画像，而不是立即要求完整参数。
3. 对未预设的新字段、新测试项或新字段族，系统仍能给出候选识别和澄清路径。
4. 系统支持至少一种复杂任务表达方式，例如字段范围、指标组、按维度分别统计、跨阶段对比。
5. 系统支持多轮对话式修订，并能继承已确认上下文。
6. 当需求有歧义时，系统会先澄清，而不是直接给出错误结果。
7. 分析结果由确定性工具生成，并能回溯字段来源、公式口径和关键步骤。
8. 系统能输出 Excel，并至少支持图表或 Markdown 报告中的一种。
9. 当模板存在时，系统能尽量兼容；当模板不适配时，系统仍能输出标准结果。
10. 发生异常时，系统能明确说明失败原因和下一步建议。

## 15. 本期结论

本项目本质上应被定义为“测试数据分析效率工具”，而不是“多智能体编排展示项目”。

真正需要被建设的能力是：

- 对复杂测试数据的理解能力
- 对自然语言分析需求的转译能力
- 对多轮修正任务的持续执行能力
- 对分析结果的解释和复核能力

如果系统最终只能跑通样例、依赖固定字段、遇到变化就失效，那么无论内部用了多少智能体，它都不能算满足本需求书。
