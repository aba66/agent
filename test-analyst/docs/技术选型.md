# 测试数据分析多智能体系统技术选型文档

## 1. 文档目标

本文档用于重新定义 `test-analyst` 项目的技术选型。

这次选型遵循以下前提：

- 使用对象主要是测试团队内部成员，规模约 `10~15` 人
- 希望直接做成可投入使用的版本，不再按“分阶段开发”写方案
- 当前不考虑大型企业级平台、中台、复杂分布式系统
- 需要保留未来升级空间，但当前架构必须轻、稳、可维护
- 明确要求采用多智能体常见技术栈：`LangGraph`、`MCP`、`Skills`、`A2A`

因此，这份文档的目标不是给出“最重、最全”的架构，而是给出“一步到位、适合小团队内部使用、同时保留升级接口”的技术方案。

## 2. 选型原则

本项目技术选型遵循以下原则：

1. 以“内部工具可用性”优先，而不是“企业级平台完整性”优先。
2. 从第一版开始就采用多智能体标准技术栈，但只落到当前规模真正需要的程度。
3. 大模型负责需求理解、会话上下文继承、澄清、工具选择、总结和协同，不负责替代全部数值计算。
4. 核心数据处理必须由程序确定性完成，保证可审计、可复核。
5. 规则配置只能作为知识补充、边界约束和回退兜底，不应替代大模型对用户真实意图的主理解。
6. 所有技术选型都优先选择 Python 生态内成熟、直观、对 10 几人团队友好的方案。

## 3. 最终推荐技术栈

一步到位推荐采用以下组合：

| 层次 | 选型 |
| --- | --- |
| 前端界面 | `Streamlit` |
| 多智能体编排 | `LangGraph` |
| 模型接入 | `OpenAI-compatible API` + `langchain-openai` |
| Agent 通信协议 | `A2A` |
| 工具接入协议 | `MCP` |
| Skills 机制 | 项目内 `SKILL.md + references + Python skill loader` |
| 数据处理 | `pandas` |
| Excel 处理 | `openpyxl` |
| 图表输出 | `Plotly` |
| 报告输出 | `Markdown + reportlab + python-docx` |
| 数据模型 | `Pydantic` |
| 本地存储 | `SQLite` |
| 配置管理 | `JSON + .env` |
| 部署方式 | 单机/单服务私有部署，保留未来 API 化升级能力 |

这套组合的核心特点是：

- 技术栈先进，但部署方式不重
- 从第一天就是多智能体方案，而不是后面再重构
- 不依赖大型企业基础设施
- 对 10 几人团队足够好维护

## 4. 核心判断

根据需求书，这个项目本质上不是通用聊天机器人，也不是企业级 AI 平台，而是一个“测试数据分析多智能体工作台”。

它的核心工作流是：

1. 用户上传一个或多个测试数据文件
2. 系统识别 sheet、字段和候选测试项
3. 用户输入自然语言需求
4. 多智能体协作完成需求解析、字段映射、规则澄清、数据分析、结果校验、报告生成
5. 输出 Excel、图表、报告和审计结果

因此，技术重点不在“做一个规则编排器”，而在：

- `LLM` 驱动的会话理解
- 结构化状态流转
- 工具调用
- 多智能体协作
- 文件与表格处理
- 人工澄清与审计

这正好对应 `LangGraph + MCP + Skills + A2A` 的组合。

## 5. 为什么要用这 4 类多智能体技术

### 5.1 `LangGraph` 作为核心编排框架

`LangGraph` 作为主干，而不是可选项。

原因：

1. 需求书本身就是一个典型的状态化工作流：会话继承 -> 文件解析 -> 需求理解 -> 澄清 -> 工具规划 -> 数据分析 -> 校验 -> 报告输出。
2. 这个流程天然需要 `state`、`interrupts`、`human-in-the-loop`、`durable execution` 这类能力。
3. `LangGraph` 不会把业务逻辑藏得太深，适合我们把会话状态、工具边界、规则兜底和 agent 协作关系控制得比较清楚。

结论：

项目从第一版开始就以 `LangGraph` 作为多智能体运行时核心。

### 5.2 `MCP` 作为工具接入标准

`MCP` 用来统一接入本项目中的工具能力，而不是所有东西都写死在主进程里。

原因：

1. 本项目有天然适合 MCP 化的能力，例如：
   - 文件读取与 sheet 摘要
   - Excel 模板写回
   - 业务规则读取
   - 报告与产物生成
2. 使用 `MCP` 后，LangGraph 中的 agent 可以通过统一方式调用工具、资源、提示，而不是每个模块都写成强耦合函数调用。
3. 未来如果要接公司内部系统、知识库、数据库，也能继续沿用 `MCP` 模式扩展。

结论：

本项目从第一版开始就用 `MCP` 组织工具层，但只建设当前必须的几个本地/私有 MCP Server，不做过度铺设。

### 5.3 `Skills` 作为 agent 能力封装方式

`Skills` 用来封装“测试分析领域能力”，不是做一个通用插件市场。

建议采用项目内技能目录形式：

- 每个技能一个目录
- 包含 `SKILL.md`
- 可选 `references/`
- 可选配套 Python 执行器

建议首批技能：

- `requirements-clarify`
- `metric-mapping`
- `decay-rule-reasoning`
- `template-fill`
- `result-audit`
- `report-summary`

原因：

1. Skills 能把测试分析领域知识从具体 agent 中拆出来，便于维护。
2. 同一技能可以被不同 agent 复用。
3. 后续新增分析规则时，不需要改整个工作流，只需要扩展技能。

结论：

项目应从第一版开始建设自己的技能层，但保持轻量，服务于测试分析场景，不追求平台化。

### 5.4 `A2A` 作为 agent-to-agent 通信协议

`A2A` 不是为了做复杂分布式集群，而是为了让本项目的 agent 协作方式从第一天就具备标准化边界。

建议做法：

1. 当前版本仍以单服务部署为主。
2. 在单服务内部，先按 `A2A` 的思想定义 agent card、task 输入输出、handoff 格式。
3. 对少数未来可能独立部署的 agent，预留 `A2A` 接口，例如：
   - 模板处理 agent
   - 业务规则 agent
   - 报告生成 agent

原因：

1. 现在用户不多，不需要把所有 agent 立刻拆成独立服务。
2. 但如果不从第一天就约定通信契约，后续升级会非常痛苦。
3. `A2A` 很适合作为“现在轻落地，未来可外扩”的协议边界。

结论：

本项目采用“单体部署 + A2A 契约优先”的方式，而不是一开始就微服务化。

## 6. 各层具体选型

### 6.1 前端层：`Streamlit`

前端继续选 `Streamlit`。

原因：

1. 当前用户规模小，交互需求主要是上传文件、输入需求、查看结果、下载产物。
2. `Streamlit` 对 Python 数据分析项目极其友好。
3. 当前没有必要上 `React + FastAPI` 这种更重的组合。
4. 如果将来真的扩到几十上百人，再考虑拆前后端也不迟。

结论：

当前前端唯一推荐选型仍是 `Streamlit`。

### 6.2 编排层：`LangGraph`

编排层直接使用 `LangGraph`。

推荐的主图结构：

1. `supervisor_agent`
2. `data_parser_agent`
3. `requirement_orchestrator_agent`
4. `metric_mapping_agent`
5. `data_analysis_agent`
6. `validation_agent`
7. `report_generator_agent`

图中必须保留：

- 状态对象
- 中断点
- 人工补充入口
- 审计 trace
- 任务恢复能力

结论：

LangGraph 是本项目的主骨架，不再用纯 Python 流程替代。

### 6.3 模型层：`OpenAI-compatible API + langchain-openai`

模型接入层建议使用：

- `OpenAI-compatible API`
- `langchain-openai`

原因：

1. 这样最容易与 LangGraph 结合。
2. 能兼容云端模型、私有模型、中转服务。
3. 后续切换模型供应商时，改动最小。

模型职责建议限定为：

- 自然语言需求理解
- 会话上下文继承与增量修订理解
- 澄清问题生成
- 工具调用计划生成
- 字段与指标语义辅助匹配
- 结果文字总结
- agent 之间的意图协商

不要交给模型的内容：

- 确定性数据计算
- Excel 精确写回
- 最终业务结论裁决

结论：

模型层采用“LLM 主导理解与工具选择 + 程序负责计算与导出”的设计。

### 6.4 MCP 层：`MCP Python SDK`

推荐使用 `MCP` 官方 Python SDK 来建设本项目的工具层。

首批建议建设 4 个 MCP Server：

1. `file-intake-mcp`
   负责文件结构识别、sheet 摘要、候选字段提取
2. `excel-template-mcp`
   负责模板 sheet 检测、样式写回、输出文件保存
3. `rules-config-mcp`
   负责字段别名、业务规则、阈值配置读取
4. `report-artifact-mcp`
   负责图表、Markdown、PDF、Docx 产物生成

当前建议：

- 先以本地/私有环境中的 `stdio` 或本地可控方式接入
- 不急着做远程 MCP 平台化部署

结论：

MCP 是工具标准化边界，不是大型平台工程。

### 6.5 Skills 层：项目内技能目录

Skills 不单独引入复杂平台，而是在项目内直接建设技能目录。

建议目录结构：

```text
skills/
  requirements-clarify/
    SKILL.md
    references/
  metric-mapping/
    SKILL.md
    references/
  decay-rule-reasoning/
    SKILL.md
    references/
  result-audit/
    SKILL.md
```

技能层职责：

- 给 agent 明确的行为边界
- 提供测试分析领域知识
- 统一澄清风格和报告风格
- 降低 prompt 散落在代码各处的问题

结论：

Skills 是本项目的知识封装层，必须从第一版就有。

### 6.6 A2A 层：轻量 Agent 通信契约

推荐把 `A2A` 用在两类地方：

1. 外部独立 agent 的通信边界
2. 内部 agent 契约建模

当前建议至少定义以下内容：

- Agent Card
- 支持的任务类型
- 输入输出 schema
- 是否支持流式结果
- 是否支持多轮交互

优先适合 A2A 化的 agent：

- `template-agent`
- `rule-agent`
- `report-agent`

结论：

当前不把系统拆成很多服务，但内部设计必须 A2A-ready。

### 6.7 数据处理层：`pandas + openpyxl`

这是本项目最不可替代的主干能力。

原因：

1. `pandas` 最适合处理 Excel/CSV 数据提取、筛选、聚合、统计和多文件合并。
2. `openpyxl` 最适合处理模板写回、sheet 结构和样式复制。
3. 需求书中的关键工作几乎全部依赖这两者。

结论：

这一层保持不变，不需要为了“AI 化”而替换。

### 6.8 图表与报告层：`Plotly + Markdown + reportlab + python-docx`

图表和报告层建议保持轻量直接：

- 图表：`Plotly`
- 中间报告：`Markdown`
- PDF：`reportlab`
- Docx：`python-docx`

原因：

1. 这是最适合内部工具的一套组合。
2. 可以快速出可交付结果。
3. 不需要复杂文档排版平台。

结论：

继续采用“结构化分析结果 + 多格式导出”的方式。

### 6.9 存储与配置层：`SQLite + JSON + .env`

当前用户量不大，因此：

- 历史记录、版本、会话、trace 用 `SQLite`
- 业务规则、别名、权限等配置用 `JSON`
- 模型配置和敏感信息用 `.env`

建议的环境变量：

- `LLM_BASE_URL`
- `LLM_API_KEY`
- `LLM_MODEL`
- `LLM_ENABLE_THINKING`
- `LANGSMITH_API_KEY`（如果后续接入）

结论：

不需要 `PostgreSQL`、`Redis`、配置中心、消息队列。

## 7. 一步到位的系统结构建议

推荐直接建设成如下结构：

```text
test-analyst/
  app.py
  run_analysis.py
  test_analyst/
    graph/
      main_graph.py
      state.py
    agents/
      supervisor_agent.py
      requirement_agent.py
      mapping_agent.py
      analysis_agent.py
      validation_agent.py
      report_agent.py
    mcp/
      file_intake_server.py
      excel_template_server.py
      rules_config_server.py
      report_artifact_server.py
    a2a/
      agent_cards.py
      schemas.py
      client.py
    skills/
      registry.py
      loader.py
    llm/
      client.py
      prompts.py
    storage/
      task_store.py
    models/
      ...
  skills/
    ...
  config/
    ...
```

这一结构的特点是：

- LangGraph 是核心
- MCP 是工具层
- Skills 是知识层
- A2A 是通信契约层
- 其他技术只为支撑这 4 层服务

## 8. 不建议采用的技术

在当前 `10~15` 人使用场景下，不建议引入：

- `FastAPI + React/Vue` 作为第一版主结构
- `PostgreSQL + Redis + Celery`
- `Kubernetes`
- `Spark / Ray`
- 向量数据库
- 企业级权限平台、SSO、中台化配置中心
- 一开始就做大量远程 agent 微服务拆分

这些技术不是永远不用，而是当前没有必要。

## 9. 对未来升级的保留方式

虽然当前不做大型企业架构，但要保留升级能力。

建议保留以下升级接口：

1. `LLM Client` 独立封装，便于切换模型服务
2. `MCP Server Registry` 独立封装，便于增加新工具
3. `Skills Registry` 独立封装，便于新增技能
4. `A2A Client/Schema` 独立封装，便于未来把某些 agent 拆成独立服务
5. 存储接口抽象，便于将来从 `SQLite` 切到 `PostgreSQL`

也就是说：

- 当前实现是轻量版本
- 但结构不是临时脚手架
- 将来升级时不需要推翻重做

## 10. 最终结论

对于 `test-analyst` 这个项目，最合适的技术选型不是“企业级 AI 平台路线”，也不是“纯 Python 规则工具路线”，而是：

`Streamlit + LangGraph + OpenAI-compatible API + MCP + Skills + A2A + pandas + openpyxl + Plotly + reportlab + python-docx + SQLite`

这是一个适合当前团队规模、同时满足你指定技术栈要求的“一步到位”方案。

它的优势在于：

- 从第一版开始就是标准多智能体架构
- 不依赖大型企业级基础设施
- 能支撑真实测试分析工作流
- 对 10 几人团队足够轻量
- 后续升级时不会推翻架构
