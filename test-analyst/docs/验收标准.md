# 测试数据分析智能体系统验收标准（对齐重构版）

## 1. 验收目标

本验收标准用于判断 `test-analyst` 是否满足新版项目需求书中定义的目标。

验收重点不是：

- 是否使用了多智能体
- 是否跑通了一个演示样例
- 是否导出了某个固定模板

验收重点是：

- 是否能处理复杂、非标准化测试数据
- 是否能通过自然语言和多轮对话完成分析闭环
- 是否能在字段变化、结构变化、需求修订时保持可用
- 是否能输出可复核、可追溯的结果

## 2. 总体验收结论规则

项目满足验收，至少需要同时满足以下条件：

1. 满足“核心能力必过项”。
2. 不触发“不可验收项”。
3. 在至少一个非样例数据场景下完成端到端分析闭环。

## 3. 核心能力必过项

### 3.1 数据理解能力

1. 用户上传测试表后，系统能够输出文件结构画像，而不是直接要求完整参数。
2. 结构画像至少包括：
   - 文件名
   - sheet 列表
   - 候选数据区或候选表头
   - 关键字段候选
   - 数据风险提示
3. 当表格存在多 sheet、多表头、空行空列、重复列名、单位混杂等常见复杂情况时，系统能够给出识别结果或澄清路径。

### 3.2 非样例适配能力

1. 系统能够处理非演示样例文件，而不是只在样例数据结构下有效。
2. 当列名、列顺序、sheet 名发生变化，但业务含义仍可识别时，系统仍能推进任务。
3. 对未预设的新字段、新测试项或新字段族，系统能够给出候选识别和人工确认路径。
4. 当字段缺失时，系统能够明确指出缺什么、影响什么，而不是静默跳过。

### 3.3 需求理解与会话继承能力

1. 系统能够像公开大模型产品一样支持连续对话。
2. 用户无需每轮都重新描述完整需求。
3. 当用户追加修订时，系统能够继承已确认上下文，例如：
   - 已选 sheet
   - 已确认字段映射
   - 已选测试项或指标组
   - 已确认公式口径
   - 已选输出方式
4. 系统能够识别“补充、修正、否定、追问”这类增量输入，而不是全部当作新任务处理。

### 3.4 澄清与收敛能力

1. 若需求存在关键歧义，系统能够先提出澄清问题，而不是直接输出错误结果。
2. 澄清问题应聚焦关键缺口，例如：
   - 目标 sheet 不明确
   - 字段映射有多个候选
   - 公式口径未定义
   - 统计维度缺失
3. 系统应避免一次性抛出过多无关问题。

### 3.5 通用分析能力

1. 系统能够按字段、测试项、指标组完成筛选、分组、聚合和统计。
2. 系统能够支持至少一种复杂任务表达方式，例如：
   - 字段范围，如 `MTFH_1~MTFH_17`
   - 指标组，如“所有亮度相关项”
   - 按某维度分别统计
   - 跨阶段对比
3. 对衰减率、变化率等业务指标，系统能够基于明确公式执行，而不是写死默认逻辑。
4. 分析结果必须由确定性工具生成，不能由模型直接编造数值。

### 3.6 多轮修订闭环能力

1. 当用户指出“结果口径不对”时，系统能够基于当前会话继续修正。
2. 当用户说“沿用刚才的口径，只改衰减公式”时，系统能够只修改对应部分。
3. 当用户要求“不要整表统计，要按测试项目分别统计”时，系统能够在继承已有上下文的基础上重新执行。
4. 多轮修订过程中，系统能够保留版本和关键上下文变化。

### 3.7 输出与交付能力

1. 系统能够输出 Excel 结果文件。
2. 系统至少支持以下一种附加交付物：
   - 图表
   - Markdown 报告
3. 当用户提供模板时，系统能够尽量兼容模板输出。
4. 当模板不适配时，系统仍能输出标准结果，而不是完全失败。
5. 输出结果默认不覆盖原始输入文件。

### 3.8 可解释与可追溯能力

1. 输出结果可追溯到原始 sheet、字段来源、公式口径和关键工具步骤。
2. 系统能够说明：
   - 使用了哪个文件和 sheet
   - 哪些字段被映射成什么业务语义
   - 使用了什么分析维度和公式
   - 哪些异常或不确定项需要人工确认
3. 审计信息至少应能落盘或在界面中查看。

## 4. 场景化验收用例

### 4.1 用例 A：陌生文件理解

验收步骤：

1. 上传一个非样例测试文件。
2. 不给出完整分析参数，只要求“先帮我看看这份表怎么分析”。

通过标准：

1. 系统能输出结构画像。
2. 系统能识别候选 sheet、候选字段和风险点。
3. 当存在歧义时，能提出合理澄清问题。

### 4.2 用例 B：复杂字段组分析

验收步骤：

1. 上传一个包含一组编号字段的测试文件。
2. 输入类似“统计 `MTFH_1~MTFH_17` 和 `MTFV_1~MTFV_17` 的最小值、最大值、平均值和衰减率，并按阶段和 SN 分别统计”的需求。

通过标准：

1. 系统能够识别字段范围或字段组。
2. 系统能够识别统计维度和统计指标。
3. 当衰减公式未定义时，系统会先澄清。
4. 明确口径后，系统能够生成结果文件。

### 4.3 用例 C：多轮修正

验收步骤：

1. 基于已有分析结果继续输入：
   - “不要整表统计，按每个测试项目分别统计”
   - “保留刚才维度，只把衰减改成相对首条记录”

通过标准：

1. 系统识别为增量修订，而不是新任务。
2. 系统能继承原有上下文。
3. 系统能输出新版本结果，并说明调整后的统计口径。

### 4.4 用例 D：模板不完全匹配

验收步骤：

1. 上传数据文件和一个模板文件。
2. 模板中字段顺序、列名或样式与当前结果不能完全一一对应。

通过标准：

1. 系统优先尝试兼容模板。
2. 若不能完全匹配，系统能明确提示不匹配点。
3. 即使模板不完全适配，系统也能输出标准结果文件。

## 5. 不可验收项

出现以下任一情况，应判定为不可验收：

1. 系统只能在演示样例上运行，换一个结构相近但字段变化的文件就失效。
2. 系统高度依赖固定字段名、固定 sheet 名或固定模板，无法处理未知结构。
3. 系统主要依赖关键词命中和规则链，无法理解多轮修订语义。
4. 用户一旦修改口径，系统就要求从头重新配置。
5. 系统在公式未明确时直接输出业务结论。
6. 系统无法说明结果是如何得到的。
7. 系统生成的关键数值不是由确定性工具计算得出。

## 6. 建议量化指标

本期建议采用以下量化目标辅助验收：

- 对 50MB 以内常见测试文件，2 分钟内完成结构画像
- 对常见单文件分析任务，5 分钟内输出首版结果
- 对同会话下的修订任务，相比首轮执行具备明显复用效果
- 至少在 3 类非样例场景中通过核心能力验收

## 7. 本期验收结论口径

只有当系统证明自己是“复杂测试数据分析效率工具”，而不是“样例驱动的流程演示系统”时，才可视为通过本期验收。
