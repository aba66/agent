# Test Analyst 使用文档

## 1. 系统简介

`test-analyst` 是一个面向测试数据分析场景的对话式分析工具。

当前版本的核心能力是：

- 上传一个或多个 `csv/xlsx/xls/xlsm` 测试数据文件
- 用自然语言描述分析需求
- 自动识别 sheet、字段、指标组、统计维度和公式缺口
- 在需要时先发起澄清，而不是直接输出错误结果
- 通过确定性程序完成筛选、分组、聚合、衰减计算和导出
- 输出 Excel、Markdown、审计日志，以及可选图表 / PDF / Docx
- 支持会话、版本和多轮修订

需要特别说明：

1. 当前系统不是“只跑样例的固定脚本”，已经支持一定程度的复杂字段组识别和上下文继承。
2. 当前系统也不是“纯规则 if-else 流程”，在配置 `LLM` 后会优先走 `LLM-first` 的需求理解；未配置时会回退到规则兼容模式。
3. 数值结果仍然由本地确定性分析代码生成，不由模型直接编造。

## 4. LLM 配置

系统支持 OpenAI-compatible 接口。

推荐以环境变量方式配置：

```bash
export LLM_BASE_URL=https://www.autodl.art/api/v1/chat/completions
export LLM_API_KEY=<your_api_key>
export LLM_MODEL=qwen3.5-397b-a17b
export LLM_ENABLE_THINKING=false
```

说明：

1. `LLM_BASE_URL` 可以填写根地址，也可以直接填写完整的 `chat/completions` 地址。
2. 当前项目对结构化 JSON 输出依赖较强，若模型支持 thinking，建议关闭。
3. 如果没有配置 `LLM_*`，系统仍可运行，但需求理解会回退到规则兼容模式。

当前版本额外兼容以下环境变量：

- `QWEN_BASE_URL`
- `QWEN_API_KEY`
- `QWEN_MODEL`
- `AUTODL_API_KEY`
- `OPENAI_API_KEY`

### 5.2 基本操作流程

1. 上传一个或多个测试数据文件。
2. 如果希望尽量贴合已有格式，可选上传模板文件。
3. 在输入框中填写自然语言需求。
4. 如果系统给出澄清问题，先补充关键信息。
5. 查看解析结果、分析结果、审计信息和输出文件。
6. 如需继续修订，直接在当前会话中继续输入补充要求。

### 5.3 页面中的可选补充项

页面侧栏可补充或覆盖以下信息：

- `LLM Base URL`
- `LLM Model`
- `LLM API Key`
- `LLM Thinking`
- `Source sheet`
- 测试项
- 阶段值
- `remark`
- 衰减公式
- 分组字段
- 输出 sheet 名称
- 模板 sheet 名称
- 阈值规则
- 图表 / Markdown / PDF / Docx 开关

这些配置是覆盖项，不是必须项。推荐优先用自然语言提出需求，仅在系统无法稳定判断时再手动指定。

## 6. 命令行使用

命令行适合做快速验证、批量运行和回归测试。

### 6.1 示例 1：样例明细输出

使用 `examples/request1.txt` 和模板文件：

```bash
cd /home/xiangdejie/projects/agent/test-analyst
python run_analysis.py \
  --file examples/input.xlsx \
  --template-file examples/output.xlsx \
  --request-file examples/request1.txt \
  --llm-base-url "$LLM_BASE_URL" \
  --llm-api-key "$LLM_API_KEY" \
  --llm-model "$LLM_MODEL" \
  --llm-disable-thinking
```

这个示例会：

- 识别 `数据汇总` sheet
- 提取 3 个指定测试项
- 按 `relative_to_first` 计算衰减
- 尝试写入 `WG-PRO-KPI Summary` 模板 sheet

补充说明：

- 当前模板导出会优先尝试保留模板格式。
- 如果模板中包含 `openpyxl` 无法直接处理的图表对象，系统会先做模板清洗再写入。

### 6.2 示例 2：字段组汇总分析

使用 `examples/request2.txt`：

```bash
python run_analysis.py \
  --file examples/input.xlsx \
  --request-file examples/request2.txt
```

这个请求默认会先触发澄清，因为用户提出了“衰减率”，但没有明确衰减公式。

如果你已经知道口径，可以直接写成：

```bash
python run_analysis.py \
  --file examples/input.xlsx \
  --request "输入excel里的数据汇总这个sheet，定位里面的MTFH_1~MTFH_17，还有MTFV_1~MTFV_17这里的数据。然后分别计算每个测试项目的MTFH和MTFV的最小值，最大值，以及平均值，还有衰减率。衰减公式按 relative_to_first 计算。"
```

当前实现会把这个请求识别为：

- 指标组：`MTFH`、`MTFV`
- 分析模式：`metric_group_summary`
- 汇总维度：`test_item`
- 统计项：`min / max / mean / decay`

### 6.3 基于已有会话继续修订

如果前一轮返回了 `session_id` 和 `task_id`，可以继续修订：

```bash
python run_analysis.py \
  --session-id <session_id> \
  --parent-task-id <task_id> \
  --request "沿用刚才的sheet和指标，衰减公式按 relative_to_first 计算。"
```

当前实现支持：

- 继承已确认的 `sheet`
- 继承已确认的测试项或指标组
- 继承已确认的统计维度
- 在 revision 中补公式而不丢失原统计口径

## 7. 示例说明

### 7.1 `examples/request1.txt`

这个示例代表“明细导出 + 模板套用”场景。

当前代码会将其识别为：

- `analysis_mode = detail`
- `group_by = ['sn']`
- 指定测试项明细输出
- 每个指标输出两列：
  - 原始值
  - 衰减值

### 7.2 `examples/request2.txt`

这个示例代表“字段组 + 统计汇总”场景。

当前代码会将 `MTFH_1~MTFH_17`、`MTFV_1~MTFV_17` 识别为字段组，并在显式提供衰减公式后，按“每个测试项目”输出统计结果。

当前实现不会再因为出现 `~` 范围表达，就默认误判为“按每个点位逐列输出”。

## 8. 输出结果说明

每次成功运行后，系统会在 `output/时间戳_任务名/` 下生成部分或全部产物：

- `analysis_output.xlsx`
  结果 Excel
- `analysis_report.md`
  Markdown 报告
- `audit_log.json`
  审计日志
- `audit_report.md`
  审计报告
- `*.html`
  图表文件
- `analysis_report.pdf`
  如果环境支持 PDF 导出
- `analysis_report.docx`
  如果环境支持 Docx 导出

## 9. 结果解释

输出结果中通常可以看到：

- 解析后的结构化需求
- 关键字段映射
- 指标匹配结果
- 统计摘要
- 审计告警
- 版本与 trace 信息

这意味着系统不仅输出文件，还尽量说明：

- 用了哪个 sheet
- 识别了哪些字段或字段组
- 用了什么公式
- 哪些步骤触发了澄清或回退

## 10. 当前限制

当前版本仍有以下限制：

1. `LLM` 未配置时，复杂自然语言理解能力会下降。
2. `PDF` / `Docx` 导出依赖额外库，未安装时会跳过。
3. 模板兼容是“尽量适配”，不是所有复杂 Excel 模板都能百分百复刻。
4. 某些极端复杂表格仍需要用户手动指定 `sheet`、字段或公式口径。

## 11. 建议使用方式

推荐按以下方式使用：

1. 先上传文件，让系统识别结构。
2. 先给自然语言需求，不要一开始就手动填满所有参数。
3. 若系统发起澄清，只补关键缺口。
4. 如果首版结果不对，直接在当前会话中继续修订，不要从头重建任务。

这也是当前项目需求书和技术设计文档定义的推荐使用路径。
