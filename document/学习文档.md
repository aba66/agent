下面是我按你当前基础重排的一版 **8 周学习与项目排期表**。原则只有三个：

第一，**先把单 agent 做稳，再碰多 agent**。
第二，**先做能跑的工程闭环，再谈生产化**。
第三，**8 周内主线只用一套核心编排思路：LangGraph 为主，其他框架按阶段补，不并行乱学**。之所以这样排，是因为现在 LangChain 的 agent 本身就是建立在 LangGraph 之上的，而 LangGraph 官方把 durable execution、persistence、human-in-the-loop、memory 都放在核心位置；这正好对应你做工程落地最需要的“可恢复、可审计、可插手”。([LangChain 文档][1])

---

## 8 周总目标

8 周结束后，你要真正做出来的，不是一个“会聊天的 agent”，而是一个 **能读配置、生成训练命令、启动实验、解析日志、输出建议、支持人工审批、保存状态、失败后可恢复** 的最小智能体工程系统。第 8 周时，再把它补成一个简化版“实验闭环平台”。这条路线会先用 LangGraph 打基础，中间补 OpenHands 风格的沙箱执行，再接 AgentScope Runtime 的服务化思路，最后接入 MLflow 和 Ray；KServe 放在第 8 周做了解和可选接入，不强行做重部署。([LangChain 文档][1])

## 第 1 周：只做“单 agent + 工具调用”最小闭环

### 本周目标

你先不要碰多智能体，不要碰平台，不要碰部署。只做一件事：
**输入一句自然语言，agent 能把它转成标准化训练参数，并生成一条可执行训练命令。**

### 这周学什么

重点只学 4 个概念：

1. 结构化输出
2. tool/function calling
3. 状态对象怎么设计
4. 一个最小 LangGraph 节点流程怎么跑起来

你这周不需要学 persistence 细节，更不需要学 memory。只要把 “用户需求 → 参数解析 → 配置检查 → 命令生成” 走通就够了。LangGraph 的核心思路就是把任务拆成离散节点，由共享 state 在节点间流转；而 Functional API 也支持你在保留普通 Python 控制流的同时，把 persistence、human-in-the-loop、streaming 这些能力逐步补进去，所以它很适合先从简单流程起步。([LangChain 文档][2])

### 本周项目

做一个 **训练命令生成助手 v1**：

输入示例：
“用 ResNet50 跑 CIFAR10，batch_size=64，epoch=50，lr=1e-3”

输出示例：

* 结构化参数 JSON
* 配置校验结果
* 最终训练命令
* 风险提示（比如 batch 太大、路径缺失）

### 每天安排

周 1-2：看 LangGraph overview、thinking in LangGraph，理解 state / node / edge。
周 3：写一个 `parse_request` 节点。
周 4：写一个 `validate_config` 节点。
周 5：写一个 `build_command` 节点。
周 6：把它们串成单图。
周 7：自己测 10 条自然语言请求，修 prompt 和 schema。([LangChain 文档][1])

### 本周交付物

你这一周结束时必须交出这 4 个东西：

* 一个 Git 仓库
* 一个可运行 demo
* 一个 `examples/` 目录，放 10 条输入样例
* 一个 `README`，说明输入、输出和已知限制

### 验收标准

满足下面 3 条就算过关：

* 10 条样例里，至少 8 条能正确解析核心参数
* 遇到缺失参数时，不是胡编，而是显式报缺失项
* 输出命令能直接复制到终端中使用

## 第 2 周：把单 agent 从“会生成命令”升级到“会执行并看日志”

### 本周目标

把第 1 周的 agent 接到你熟悉的训练脚本上。
变成：**读需求 → 生成命令 → 调脚本 → 抓前几行日志 → 输出简短训练摘要**

### 这周学什么

重点学：

1. tool 节点真正调用 Python / shell
2. stdout / stderr 采集
3. 超时与重试
4. 错误分类（配置错误、路径错误、环境错误）

这一步本质上还是单 agent，只是工具从“返回命令字符串”变成“执行命令并收集反馈”。

### 本周项目

把项目升级成 **自动化实验助手 v2**：

新增能力：

* 运行本地脚本
* 抓取日志前 N 行
* 识别常见启动失败
* 返回“成功启动 / 启动失败 + 原因”

### 每天安排

周 1-2：把原先命令生成功能抽成 tool。
周 3：加 `run_command`。
周 4：加日志截断与错误分类。
周 5：加重试规则，只允许对“瞬时错误”重试。
周 6-7：联调你自己的一个简单训练仓库。

### 本周交付物

* `tools/run_command.py`
* `logs/` 自动保存
* 启动成功与启动失败两类示例
* 一份错误码约定文档

### 验收标准

* 能成功启动至少 1 个真实训练脚本
* 能把失败分成至少 3 类
* 不允许 agent 在失败时编造“训练已开始”

## 第 3 周：补上你后面一定会用到的“持久化、审批、恢复”

### 本周目标

把单 agent 做成一个 **可中断、可恢复、可人工审批** 的流程。
这是 8 周里最关键的一周，因为这一步会把你的思路从“脚本调用”切到“工程工作流”。

### 这周学什么

重点补：

1. persistence / checkpoint
2. human-in-the-loop
3. thread / run state
4. 中断后恢复

LangGraph 官方明确把 durable execution 和 persistence 作为一等能力：工作流可以在关键点保存进度，暂停后再从原位置恢复；这尤其适合 human-in-the-loop 场景，也适合长任务失败后继续执行。([LangChain 文档][3])

### 本周项目

升级为 **带审批的实验启动器 v3**：

流程变成：
需求输入 → 参数解析 → 配置检查 → 生成命令 → **等待人工审批** → 真正执行 → 保存状态

### 每天安排

周 1：看 durable execution / persistence 文档。
周 2：给 graph 接 checkpointer。
周 3：增加 `approve_run` 人工节点。
周 4：实现暂停与恢复。
周 5：模拟中途崩掉再恢复。
周 6-7：整理状态字段，只保留必要项。([LangChain 文档][3])

### 本周交付物

* 支持审批的工作流图
* 一个 `state_schema.py`
* 一个恢复测试脚本
* 一份状态流转图

### 验收标准

* 审批前不会真正启动训练
* 中途中断后可以恢复
* 每次运行都有唯一 run_id / thread_id
* 能重放上一次状态，不靠口头回忆

## 第 4 周：进入多智能体，但只准 3 个角色

### 本周目标

从单 agent 升级到 **三角色多智能体闭环**。
注意，不要加更多角色。你现在最需要学的是“职责切分”，不是“角色堆叠”。

### 角色固定为

* **Planner**：理解需求，产出实验计划
* **Executor**：执行训练与收集日志
* **Evaluator**：看结果，给出下一轮建议

### 这周学什么

重点是：

1. 共享 state 怎么设计
2. 节点职责边界
3. 条件分支
4. 一轮结束后是否继续下一轮

LangGraph 的多节点状态图模型很适合做这种分工式流程，因为每个节点都读写共享状态，再通过条件边决定流向。([LangChain 文档][2])

### 本周项目

做一个 **实验闭环三智能体 v1**：

要求只跑 2~3 轮，不追求自动调得多聪明，只要求闭环完整：

* Planner 读目标
* Executor 跑实验
* Evaluator 看指标
* 若未达标，则回到 Planner 生成下一轮参数

### 每天安排

周 1：先画状态图。
周 2：拆 3 个 agent 的输入输出。
周 3：实现 Planner。
周 4：实现 Evaluator。
周 5：串 Executor。
周 6-7：跑通 2 轮自动实验。

### 本周交付物

* 一张完整状态图
* 2~3 轮实验闭环日志
* 每轮结果 JSON
* 一份“为什么继续 / 为什么停止”的规则说明

### 验收标准

* 多轮流程能自动结束，不死循环
* Evaluator 的判断有明确规则，不是纯口语
* 每轮实验结果都能落盘保存

## 第 5 周：学执行层，但只选一个方向先做

### 本周目标

把 agent 从“调用你预留好的工具”升级成“能在隔离环境里进仓库做事”。
你这周只在 **OpenHands** 和 **DeerFlow** 里选一个方向先了解，但项目建议优先按 **OpenHands 的 sandbox 思路** 来练，因为它更贴近“代码仓库执行、命令隔离、文件操作”这类工程动作。

OpenHands 官方文档当前仍把 Docker sandbox 作为本地运行时的默认且推荐选项，并强调容器隔离；还有更偏 SDK 的 DockerWorkspace 方案，也强调对宿主机的完整隔离，适合生产部署、测试和安全执行不可信代码。([docs.openhands.dev][4])

### 这周学什么

重点是：

1. workspace
2. sandbox
3. 文件系统读写
4. shell 命令执行
5. traceback 捕获与固定模板修复

### 本周项目

做一个 **训练仓库执行 agent**：

要求它能：

* 打开指定 repo
* 检查依赖文件
* 运行训练命令
* 捕获报错
* 自动修复 1~2 类固定错误

只修最简单的：

* 路径写错
* 输出目录不存在
* 配置文件缺失字段

### 每天安排

周 1：看 OpenHands sandbox 文档，理解隔离边界。
周 2：为你的项目写一个本地 Docker 运行环境。
周 3：让 agent 在 sandbox 中读 repo。
周 4：加入命令执行。
周 5：加入报错解析。
周 6-7：实现两个固定错误的自动修复。([docs.openhands.dev][4])

### 本周交付物

* 一个 Docker 化的实验仓库环境
* `fix_path_error` 和 `fix_output_dir_error` 两个自动修复器
* 一份 sandbox 边界说明

### 验收标准

* agent 能在隔离环境中执行命令
* 修复动作仅限白名单类型
* 修复后会重新执行并报告结果
* 不允许 agent 任意改仓库文件

## 第 6 周：开始服务化，但只做“最小平台”

### 本周目标

让你前面做的三角色系统，不再只是本地脚本，而是一个 **有任务 ID、有状态、有日志查询能力的最小服务**。

这一周建议引入 **AgentScope Runtime 的思路**，但不是让你重写全部系统，而是借它的工程视角：secure tool sandboxing、Agent-as-a-Service APIs、scalable deployment、full-stack observability。它的价值不在“替代你前面的 LangGraph”，而在帮助你建立“agent 是服务，不是脚本”的思维。([GitHub][5])

### 这周学什么

重点是：

1. task_id / run_id
2. API 化
3. 日志查询接口
4. 状态查询接口
5. 最简单的 trace 记录

### 本周项目

做一个 **实验闭环平台最小版**：

最少提供 4 个接口：

* 创建实验任务
* 查询任务状态
* 查看最新日志
* 恢复失败任务

### 每天安排

周 1：读 AgentScope Runtime README，理解它强调的服务化能力。
周 2：给你现有系统套 FastAPI。
周 3：把状态存进数据库或文件存储。
周 4：增加日志查询。
周 5：增加恢复接口。
周 6-7：做一次本地压测和异常恢复测试。([GitHub][5])

### 本周交付物

* 一个最小 API 服务
* 简单任务状态页或命令行查询器
* 一份接口文档
* 一份恢复测试报告

### 验收标准

* 一个任务可通过 API 创建
* 可查询执行中 / 成功 / 失败三种状态
* 失败任务可恢复
* 日志能按 task_id 查到

## 第 7 周：补上真正和算法工程最相关的 MLOps 闭环

### 本周目标

把 agent 跟 **实验追踪、模型注册、批量调参** 接起来。
这一周是你从“会做 agent”走向“agent 真能解决算法工程问题”的关键。

MLflow Tracking 现在仍然把自己定义为用于记录参数、代码版本、指标和产物文件的 API 和 UI；Model Registry 则是集中管理模型生命周期、版本、别名和 lineage 的中心化模型仓库。Ray Tune 则是可做大规模实验执行和超参搜索的 Python 库，支持多种调参算法。([MLflow AI Platform][6])

### 这周学什么

重点是：

1. MLflow Tracking
2. MLflow Registry
3. Ray Tune
4. agent 怎样触发批量实验
5. agent 怎样比较 run 并选择最优模型

### 本周项目

做一个 **MLOps 闭环最小版**：

流程：

* Planner 生成调参计划
* Executor 调用训练
* 每次 run 写入 MLflow
* Evaluator 读取 MLflow 结果比较
* 把最优模型注册到 Registry
* 用 Ray Tune 跑一个小规模搜索

### 每天安排

周 1：给现有训练脚本接 MLflow Tracking。
周 2：让 agent 能查询历史 runs。
周 3：接入 Model Registry。
周 4：学 Ray Tune 的最小搜索示例。
周 5：把 Planner 改成能生成搜索空间。
周 6-7：跑一个小型调参实验。([MLflow AI Platform][6])

### 本周交付物

* 一个可用的 MLflow 实验面板
* 至少 1 个注册模型
* 1 次 Ray Tune 小型搜索
* 一份“为什么选这个最优模型”的自动报告

### 验收标准

* 每轮实验都有参数、指标、产物
* agent 能基于历史结果给出下一轮建议
* 最优模型可被注册而不是只保存在本地文件夹
* Ray Tune 至少能并行或批量跑多个候选配置

## 第 8 周：系统整合与可选部署出口

### 本周目标

把前 7 周内容整成一个 **最小可演示系统**。
这周不要再大幅扩框架，只做整合、减法、打磨、复盘。

### 这周必须完成的主线

把系统串成下面这条完整链：

需求输入 → Planner 生成实验计划 → 人工审批 → Executor 启动实验 → 日志与状态持久化 → Evaluator 看结果 → MLflow 记录 → 选择最优模型 → 输出总结报告

### KServe 怎么处理

KServe 现在官方仍把自己定位为运行在 Kubernetes 上、支持多框架可扩展部署的推理平台，并强调 autoscaling、canary、monitoring、revision / rollout 管理。
但对你这 8 周来说，**KServe 只需要做到“了解作用 + 画接入位 + 做可选 PoC”**，不建议强行把 Kubernetes 全套压进这 8 周主线里，否则你会从“学 agent 工程”变成“学云原生运维”。([KServe][7])

### 这周可选项

你可以二选一：

* **轻量选项**：只把最优模型导出，保留本地 serving 或简单 API
* **进阶选项**：额外做一个 KServe PoC，理解 InferenceService、canary rollout、回滚思路

### DeerFlow 放哪

DeerFlow 官方当前强调的是 long-horizon、super agent、sub-agents、memory、sandboxes，更适合分钟到小时级的长任务总控。它更像你后续把整条链再“总控化”的放大器，而不是这 8 周主线里的必需品。
所以第 8 周你最多做两件事之一：
要么读 DeerFlow 架构，理解它为什么适合更长链任务；
要么不碰 DeerFlow，专心把自己的最小系统打磨好。([GitHub][8])

### 本周交付物

* 一个完整 demo
* 一份架构图
* 一份 5~10 分钟演示脚本
* 一份复盘文档：哪些是稳定能力，哪些还只是实验能力

### 验收标准

* 一条完整链能跑通
* 至少一次真实实验可被端到端追踪
* 失败后可恢复
* 最终结果能被归档和解释
* 演示时不需要你手工改 10 个地方才能成功

## 你这 8 周明确不要做的事

* 不要同时学 LangGraph、AutoGen、LlamaIndex、CrewAI、Dify、AgentScope 全家桶。
* 不要一开始就做 5 个以上 agent。
* 不要一上来搞“自动改模型结构、自愈、全自动上线”。
* 不要把大量时间花在调基座模型、提示词花活和酷炫前端上。
* 不要把 K8s / KServe 当成第 1-4 周的主线。

你的主线一直应该是：
**状态化工作流 → 可恢复执行 → 受控执行 → 服务化 → 实验追踪 → 调参与模型管理。**
这个顺序最贴合你“有深度学习基础、第一次接触智能体开发、目标是工程落地”的起点。([LangChain 文档][1])

## 最后给你一个最实际的执行建议

你现在不要想着“8 周后我要做超级智能体”。
你真正该盯住的是这 3 个里程碑：

* **第 2 周末**：能自然语言启动一个真实训练
* **第 4 周末**：能自动跑完 2~3 轮实验闭环
* **第 7 周末**：能把实验记录、模型选择、调参闭环接到 MLflow + Ray 上

只要这 3 个里程碑都落地，你这 8 周就已经学得很值了。接下来最值得做的是，先把 **第 1 周的仓库骨架和目录结构** 搭起来。
